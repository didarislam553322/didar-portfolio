Evaluating AI Feature Proposals: A Strategic Checklist

Strategy 3 min read · Didar Islam

AI features show up in nearly every roadmap discussion now, often because a competitor has one, not because users asked for it. Here are six questions to ask before committing engineering time to one.

To keep things concrete, imagine the same proposal running through all six: a team wants to add an "AI assistant" to a project management tool, one that reads a project and writes a status update for you.

1. What problem does this solve, without using the word "AI"?

Here's the trap: it's easy to get excited about a technology and start building before anyone checks if it actually fixes something. "Let's add an AI assistant" sounds exciting, but it doesn't say what's broken for the user. It names a piece of technology, not a problem the user actually has.

So do this test: write down the problem in one sentence, without using the word "AI" anywhere in it. If you can do that, you have a real problem. If you can't, you don't have a problem yet, you have a piece of technology someone wants to build.

Example: "Add an AI assistant" fails the test, it only names a tool. "Managers spend 40 minutes every Friday writing status updates by hand" passes the test, it names an actual pain, with a number attached, that you could go verify by asking five managers if it's true for them.

Why this matters: skip this test, and here's what actually happens. Say the team spends two months building the AI assistant anyway. It goes live and is released to all users ("ships" is the common word for this in product teams). Three months later, someone checks the usage data and finds that only 8 out of 500 managers ever clicked "generate update." The other 492 kept writing their updates by hand, the old way, because it turns out the 40-minute task was never their real complaint. Two months of engineering time went into a feature almost nobody opens, and the team only found that out after building it, not before.

2. Can this workflow handle an answer that's sometimes wrong?

AI tools are probabilistic, meaning the same input can give you a different answer each time, and mistakes are part of the deal, not a bug you'll eventually fix.

For the status update assistant, this is low-risk: if the AI writes "the team shipped 4 features" instead of 5, the manager reads it, notices, and fixes it before sending. Nobody gets hurt.

Now imagine the same AI logic used to auto-approve expense reports instead. A 5% error rate there means 1 in 20 approvals is wrong, and if the tool processes 1,000 reports a day, that's 50 bad approvals a day, with real money attached. Same error rate, completely different risk. Match the feature to a workflow where a wrong answer is cheap to catch, not one where it's expensive to miss.

3. Do the costs still make sense once this scales up?

A normal software feature, like a new filter or a new button, costs almost nothing per use once it's built. AI features are different: every single use costs real money, because each request runs through a paid model.

Let's return to the status update assistant example. Say each generated update costs 4 cents to produce, and each manager generates one update per week, four per month. That's 16 cents a month per user, cheap. But now say the team adds "generate a draft for every task, every day" instead of once a week. Now it's 30 requests a month per user instead of 4, or $1.20 a month per user. If the whole plan only costs $10 a month, that one feature just ate 12% of revenue on its own.

The lesson: before building, calculate cost per user per month, and check that number against what you actually charge. If nobody's done that math, the proposal isn't ready to build yet.

4. What makes this hard for a competitor to copy?

If your "AI assistant" is just a thin layer over a public AI model, like ChatGPT with a nice interface, then any competitor can copy it in a weekend, because they can license the same model you did.

What's harder to copy is data or context nobody else has. In the status update example: an assistant that just summarizes generic text is copyable by anyone. An assistant that pulls from your actual project timeline, task history, and who's blocked on what, all data that lives inside your specific tool, is much harder for a competitor to replicate, because they don't have access to that data.

Ask yourself: if a competitor launched the exact same feature next month, would it actually work as well for them? If yes, you don't have an advantage, you have a head start, and head starts run out.

5. How will you know if it's working, after launch?

A normal feature is done when it passes testing and ships. AI features keep changing after launch, because usage patterns shift and the underlying model itself gets updated by its provider.

For the status update assistant, "done" isn't enough. You need an actual plan: check weekly how often managers edit the AI's draft before sending it. If 90% of managers are rewriting half the update, the feature isn't working, even though it technically "launched successfully." Without that check, you won't find out it's broken until managers quietly stop using it.

6. What exactly will marketing say about it?

The claim needs to be specific enough that it's still true after people actually use the product.

"AI-powered status updates" is not a claim, it's a label, and every competitor will use the same one within a year. "Cuts your Friday status update from 40 minutes to 5, tested with 200 real managers" is a claim you can prove, defend, and that a customer can actually evaluate before buying.

Bottom Line

A proposal is ready for the roadmap once it names a real problem you've verified with users, fits a workflow where a wrong answer is cheap, has a cost-per-user number that still works at scale, relies on something a competitor cannot easily copy, has a real plan for being checked after launch, and can back up its marketing claim with a number. Very few ideas survive all six questions, which is exactly why it's worth asking them before you build, not after.
