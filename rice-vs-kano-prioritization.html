A Comparative Analysis of RICE and Kano Prioritization Frameworks

Frameworks 4 min read · Didar Islam

Feature prioritization is just the process of deciding which item on your backlog gets built first. Two popular tools handle this differently: RICE gives you a number, and the Kano model sorts features into categories based on how much they actually matter to customers. This post explains both.

To keep this concrete, one example runs through the whole post: an accounting software company deciding what to build next.

RICE: Turning a Decision Into a Number

RICE stands for Reach, Impact, Confidence, and Effort. It was built at Intercom as a way to score backlog items.

  • Reach: how many users this affects in a given time period. Example: 2,000 users per quarter.
  • Impact: how much it affects each user, on a simple scale (3 = massive, 2 = high, 1 = medium, 0.5 = low, 0.25 = minimal).
  • Confidence: how sure you are about your Reach and Impact numbers, as a percentage. Use 100% if both numbers come from real usage data or a real experiment. Use 80% if one number is backed by data and the other is an educated guess. Use 50% if both numbers are mostly opinion.
  • Effort: how much work it takes, measured in person-months (one person working for one month equals one person-month).

The formula: RICE score = (Reach x Impact x Confidence) / Effort

Here's the accounting company's example: a new "export dashboard to PDF" feature. Reach is 2,000 users a quarter. Impact is 1 (medium). Confidence is 80%, since Reach comes from real usage logs but Impact is a designer's estimate. Effort is 2 person-months.

RICE score = (2,000 x 1 x 0.8) / 2 = 800

If a competing feature scores 400, the export feature wins the ranking. Tools like Jira, Productboard, or Aha! let you track these scores directly on your backlog.

Kano: Sorting Features by What They Actually Do for Users

The Kano model, created by Professor Noriaki Kano in 1984, sorts features into three groups based on how they affect customer satisfaction.

Basic features. Users expect these by default. Having them adds no extra happiness, but missing them makes people angry. In the accounting product: password reset, exporting your data, and the app actually staying online. Nobody praises these in a review, but losing them makes customers leave.

Performance features. The better these are, the happier users get, in a straight line. In the same product: how fast reports generate, and how many banks the software connects to. Ten bank connections make users happier than five, and twenty make them happier than ten.

Delighters. Unexpected extras that make people happy, but nobody's upset if they're missing. Example: an alert that catches a weird, unusual expense before the customer even notices it themselves. Nobody asked for this, but it's the kind of feature people mention by name in reviews.

You figure out which category a feature falls into by surveying users with two questions: how would you feel if this existed, and how would you feel if it did not.

When to Use Which One

Kano works best early on, when you're not trying to rank features yet, just trying to figure out what kind of value each one provides. A company entering an already crowded market (a new project management tool, for example) should use Kano first, to make sure it has every basic feature competitors already have, before spending time on anything fancy. A company that's already the market leader should use Kano to find delighters instead, since the basics are already assumed by that point.

Kano also changes over time. Cloud file syncing felt like a delighter back in 2008. Today it's just a basic feature everyone expects. That's why it's worth re-running the Kano survey every so often instead of treating the results as permanent.

RICE works best once you already have a list of candidate features and need a defensible way to rank them using real numbers. Its weakness is that a RICE score is only as good as the estimates that went into it. Weak guesses produce a precise-looking number that is still just a guess.

Bottom Line

Use Kano first, to make sure the roadmap covers the basics and the performance features before chasing anything flashy. Then use RICE within each category to decide what order to actually build things in. Used together, you get a backlog that's both balanced across the right kinds of value, and ranked by real numbers within each one.
